# Foreman — Parser & Graph Load Technical Brief

This document describes the design and implementation requirements for:

1. The `en-US.json` parser in `packages/mcp/src/parser/`
2. The Kùzu graph load pipeline in `packages/mcp/src/graph/`

---

## Source File

**Path (Steam, 1.x):** `<game install>/CommunityResources/Docs/en-US.json`
**Path (Epic, 1.x):** `<game install>/CommunityResources/Docs/en-US.json`
**Pre-1.0:** The file was named `Docs.json` in the same directory. Multi-version support should handle both filenames.

The file is auto-generated by Coffee Stain Studios when the game is compiled and ships with every update. It is the canonical source of truth for all game data used by Foreman.

---

## Known Parsing Challenges

These are real gotchas documented by the community. Handle all of them.

### 1. UTF-16 LE with BOM
The file is encoded in **UTF-16 Little Endian** with a **byte order mark (0xFEFF)** at the start. Standard `fs.readFileSync` with no encoding specified returns a Buffer — decode it as UTF-16 LE and strip the BOM before passing to `JSON.parse`.

```typescript
const buffer = fs.readFileSync(filePath);
const content = buffer.toString('utf16le').replace(/^\uFEFF/, '');
const raw = JSON.parse(content);
```

### 2. Top-Level Structure
The file is an array of class group objects:

```json
[
  {
    "NativeClass": "Class'/Script/FactoryGame.FGItemDescriptor'",
    "Classes": [ ... ]
  },
  {
    "NativeClass": "Class'/Script/FactoryGame.FGRecipe'",
    "Classes": [ ... ]
  }
]
```

Each `NativeClass` string identifies the category. Extract the short class name using the pattern `\.(\w+)'$`.

### 3. No Exposed Class Hierarchy
There are 30+ classes all extending `FGItemDescriptor` (and other base classes) with no hierarchy surfaced in the file. Maintain a manual mapping from `NativeClass` short name → internal category:

| Short Name | Category |
|---|---|
| `FGItemDescriptor` | item |
| `FGItemDescriptorBiomass` | item |
| `FGItemDescriptorNuclearFuel` | item |
| `FGResourceDescriptor` | resource |
| `FGEquipmentDescriptor` | equipment |
| `FGConsumableDescriptor` | item |
| `FGRecipe` | recipe |
| `FGBuildable` | building |
| `FGBuildableManufacturer` | building |
| `FGBuildableResourceExtractor` | building |
| `FGBuildableWaterPump` | building |
| `FGBuildableFrackingExtractor` | building |
| `FGBuildableGeneratorFuel` | building |
| `FGBuildableGeneratorNuclear` | building |
| `FGSchematic` | schematic |

Any unrecognised `NativeClass` should be logged (not thrown) and skipped.

### 4. Custom Ingredient/Product Encoding
`mIngredients` and `mProduct` fields are **not standard JSON arrays**. They use a custom Unreal Engine string format:

```
((ItemClass=BlueprintGeneratedClass'...Path.ClassName_C',Amount=100),(ItemClass=...,Amount=50))
```

**Parsing steps:**
1. Strip the outer `(( ... ))` — remove first 2 and last 2 characters
2. Split on `),(` to get individual entries
3. For each entry, regex-match:
   - `ItemClass`: `[^']+\'[^.]+\.(\w+)\'` → capture the class name (e.g. `Desc_IronPlate_C`)
   - `Amount`: `Amount=(\d+)` → parse as integer

```typescript
function parseIngredients(raw: string): { className: string; amount: number }[] {
  if (!raw) { return []; }
  const inner = raw.slice(2, -2);
  return inner.split('),(').map(entry => {
    const classMatch = entry.match(/[^']+\'[^.]+\.(\w+)\'/);
    const amountMatch = entry.match(/Amount=(\d+)/);
    return {
      className: classMatch?.[1] ?? '',
      amount: parseInt(amountMatch?.[1] ?? '0', 10),
    };
  }).filter(i => i.className !== '');
}
```

### 5. String-Encoded Numbers
Several numeric fields are stored as strings and must be parsed:
- `mManufactoringDuration` → `parseFloat` (note: deliberate typo in source — match it exactly)
- `mQuantity` → `parseInt`
- `mStackSize` → `parseInt`

### 6. Missing Cross-References
`FGBuildDescriptor` does not include `mBuildableClass`, making it impossible to directly link the build item to its `FGBuildable` entry. Use name-based heuristics: strip `Build_` prefix and `_C` suffix, or maintain a small manual mapping for known exceptions.

### 7. Build Costs vs Production Recipes
These are distinct and must be kept separate:
- **Build costs** come from `FGBuildDescriptor` build-gun recipes — what it costs to place a building
- **Production recipes** come from `FGRecipe` — what a building produces

Do not conflate them. The graph models them as separate relationship types (`BUILD_COST` vs `CONSUMES`/`PRODUCES`).

## Edge Cases — Handle, Don't Crash

- **Alternate recipes** — an item usually has several. The `isAlternate` flag is present on `FGRecipe`. Preserve all of them; surface them in the `recipes_for` tool.
- **Fluids and gases** — measured in m³/min, not items/min. The `mForm` field on items distinguishes `solid`, `liquid`, and `gas`. Amounts in fluid recipes are in units of 1000 = 1 m³; divide by 1000 when computing per-minute rates for display.
- **Variable-power recipes** — some buildings (Particle Accelerator, Converter, Quantum Encoder) have power consumption that varies by recipe. Store the recipe-level power value, not just the building-level value.
- **Raw resources with no producing recipe** — iron ore, water, crude oil etc. have no `FGRecipe` that produces them. They are leaf nodes in the graph. `total_raw_inputs` terminates at these nodes.
- **Recipes with byproducts** — some recipes produce multiple outputs (e.g. Rubber + Heavy Oil Residue). All outputs appear in `mProduct`. Model all of them as `PRODUCES` relationships; do not discard secondary outputs.
- **Items with no recipe** — valid. Surface them but don't error.

---

## Computed Values at Ingest Time

Compute these during parsing and store them — do not recalculate at query time:

```
perMinute = amount * 60 / durationSeconds
```

Store `perMinute` on each `PRODUCES` and `CONSUMES` relationship in the graph. This makes production rate queries trivially cheap.

## Parser Architecture

**Single responsibility:** the parser converts the raw file into clean `GameData`. It has no knowledge of MCP, work orders, or the foreman. All Unreal Engine string noise is resolved here so nothing upstream ever sees it.

```
packages/mcp/src/parser/
  index.ts              ← public API: parseDocsFile(filePath: string): ParseResult
  reader.ts             ← UTF-16 BOM handling, file read
  classMap.ts           ← NativeClass string → category mapping
  types.ts              ← all TypeScript interfaces (GameData, Item, Recipe, Building, Schematic)
  extractors/
    items.ts            ← handles all FGItemDescriptor subclasses
    recipes.ts          ← FGRecipe: ingredients, products, duration, machine, isAlternate
    buildings.ts        ← FGBuildable subclasses + build costs
    schematics.ts       ← FGSchematic: milestones, MAM, AWESOME Shop, hard drives
    resources.ts        ← FGResourceDescriptor: raw resources
  normalise/
    ingredients.ts      ← parses the ((ItemClass=...,Amount=...)) format
    classRef.ts         ← resolves Unreal class name strings to clean short names
    fluids.ts           ← converts fluid amounts (÷1000) and labels units correctly
```

## Graph Load Architecture

The graph layer sits between the parser and the MCP tools. It is responsible for loading `GameData` into Kùzu and providing a clean query interface.

```
packages/mcp/src/graph/
  index.ts              ← public API: initGraph(gameData: GameData): GraphDB
  schema.ts             ← Kùzu DDL: CREATE NODE TABLE, CREATE REL TABLE
  loader.ts             ← idempotent load from GameData (drop + reload on rebuild)
  queries/
    items.ts            ← get_item, what_consumes, what_produces
    recipes.ts          ← get_recipe, recipes_for, compare_alternates
    production.ts       ← ingredient_tree, total_raw_inputs, buildable_with
    schematics.ts       ← list_schematics, get_schematic
  types.ts              ← query result types (distinct from parser types)
```

**Kùzu notes:**
- Embedded, in-process — no daemon, no network, no infra. Import as a Node.js package.
- Relationship tables are typed. Model `UNLOCKS` as three separate tables: `UNLOCKS_RECIPE`, `UNLOCKS_BUILDING`, `UNLOCKS_ITEM`.
- Load is idempotent: drop all tables and reload on every rebuild. The dataset is small enough that a full rebuild is the right strategy.

## Output Types

```typescript
interface ParseResult {
  gameData: GameData;
  parseWarnings: string[];   // non-fatal issues logged during parse
}

interface GameData {
  version: string;           // detected from file metadata, or 'unknown'
  parsedAt: string;          // ISO timestamp
  items: Record<string, Item>;
  resources: Record<string, Resource>;
  recipes: Record<string, Recipe>;
  buildings: Record<string, Building>;
  schematics: Record<string, Schematic>;
}

interface Item {
  className: string;
  displayName: string;
  description: string;
  stackSize: number;
  form: 'solid' | 'liquid' | 'gas' | 'invalid';
  sinkPoints: number;
}

interface Resource extends Item {
  isResource: true;
}

interface Ingredient {
  itemClassName: string;
  displayName: string;       // resolved at parse time
  amount: number;
  perMinute: number;         // computed at parse time
  unit: 'items' | 'm³';     // derived from item form
}

interface Recipe {
  className: string;
  displayName: string;
  isAlternate: boolean;
  craftTime: number;         // seconds (parsed from mManufactoringDuration)
  ingredients: Ingredient[];
  products: Ingredient[];
  producedIn: string[];      // display names of machines
  inBuildGun: boolean;
  inWorkshop: boolean;
  variablePower?: { min: number; max: number }; // MW, for variable-power buildings
}

interface Building {
  className: string;
  displayName: string;
  description: string;
  category: string;
  powerConsumption: number;  // MW baseline
  buildCost: { itemClassName: string; amount: number }[];
}

interface Schematic {
  className: string;
  displayName: string;
  type: 'milestone' | 'mam' | 'awesome_shop' | 'hard_drive' | 'tutorial' | 'other';
  tier: number;
  cost: Ingredient[];
  unlocksRecipes: string[];
  unlocksBuildings: string[];
  unlocksItems: string[];
}
```

## Version Detection

Attempt to detect the game version from the file or its surrounding context:
1. Check for a version field in the file metadata (if present)
2. Check for a `GameVersion.txt` adjacent to the `Docs` folder (community convention)
3. Fall back to `'unknown'` — never throw

---

## Error Handling Philosophy

- **Never throw on a bad entry.** Log a warning and skip it.
- **Never silently discard data without a trace.** All skipped entries are collected into `parseWarnings[]`.
- **A partial parse is better than no parse.** If 95% of recipes are clean, surface them and warn about the 5%.

## Testing

Write unit tests (Vitest) for:

- `reader.ts` — BOM stripping and UTF-16 decoding
- `normalise/ingredients.ts` — all edge cases of the custom encoding format, including fluids and byproducts
- `normalise/classRef.ts` — class name resolution
- `extractors/recipes.ts` — full recipe extraction, `isAlternate` detection, `producedIn` resolution
- `graph/queries/production.ts` — `ingredient_tree` and `total_raw_inputs` against known values

**Known-good test values:**
- Reinforced Iron Plate (standard and alternate recipes)
- Turbo Motor (deep dependency tree, good for `ingredient_tree`)
- Rubber (byproduct recipe — also produces Heavy Oil Residue)
- Water (raw resource — no producing recipe, leaf node)

Use hand-crafted fixture JSON (not the real game file) that covers all known edge cases. Do not commit `en-US.json` to the repo.
