# Contributing to FICSIT Foreman

Thanks for helping out. This is a community-first project ÔÇö contributions of
code, docs, and game-data updates are all welcome.

## Conventions

This file is the single home for our conventions (`CLAUDE.md` points here).

### Commits ÔÇö Conventional Commits, body required
- Format `type(optional-scope): subject` ÔÇö imperative, lowercase subject, no
  trailing full stop (e.g. `feat(server): add HTTP transport`).
- Allowed types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`,
  `build`, `ci`, `chore`, `revert`.
- **A body is required** ÔÇö explain *what* changed and *why* (wrap ~72 cols).
- Breaking changes: `type!: subject` plus a `BREAKING CHANGE:` footer.

### Branches
- Prefix by work type: `feature/<slug>`, `bugfix/<slug>`, `hotfix/<slug>`.
- `<slug>`: lowercase, words joined by `-`, drop filler words.
- **Never commit directly to `main`** ÔÇö branch first (also enforced by a ruleset).

### Pull requests
- One logical change per PR; keep it small (aim < ~500 changed lines).
- **Squash-merge** so `main` stays linear; the squash message follows the commit
  convention above.
- Update a branch from `main` with **rebase**, not a merge commit.
- Title: the Conventional Commit summary. Write the **complete** description at
  creation time, not in a later edit.
- AI-generated PRs must begin the body with: `­şñû AI-generated PR ÔÇö Please review
  carefully.`

### Code standards
- Always use curly braces, even for single-line `if`/`else`/loops.
- Explicit return types on all exported functions.
- No `any` ÔÇö use `unknown` and narrow it.
- Errors are logged with context, never silently swallowed.
- Parser warnings are collected into `parseWarnings[]`, not thrown.
- **British English** in all comments and documentation.

## Development

```bash
npm install
npm run build      # type-check / build the workspaces
npm run lint       # ESLint + Prettier
npm test           # Vitest
```

CI (`build` + `lint` + `test`) runs on every PR and must pass before merge.
Docker image builds run only when image-relevant files change.

## Supplying game data (the channel bundle)

The MCP servers can read game data from a local install, but the app also ships
**bundled game data per release channel** so it works out of the box. At most two
channels are kept ÔÇö the latest **stable** and the latest **experimental** build.
Each channel is a **three-file bundle** that all describes the *same* game build:

```
packages/game-data-core/data/
  stable/        en-US.json + meta.json + world-locations.json
  experimental/  en-US.json + meta.json + world-locations.json
```

- **`en-US.json`** ÔÇö the raw Satisfactory docs export (game data).
- **`meta.json`** ÔÇö `{ "gameVersion": "1.2.3.1", "build": 495413, "channel": "stable" }`.
- **`world-locations.json`** ÔÇö the extracted collectible/resource-node dataset
  (coordinates, purity, counts), regenerated by
  [`tools/world-locations`](./tools/world-locations/README.md).

When you update a channel, follow these rules ÔÇö they are enforced by the
**Validate game data** CI check:

> **One channel per PR, as a complete bundle.** A data PR must contain **only**
> files under a single `packages/game-data-core/data/<channel>/` directory (no code,
> no docs, no other channel), and that channel must end up with **all three** files
> present.

> **Never bump a version without re-extracting.** `meta.json` and
> `world-locations.json` must agree on **both** `gameVersion` and `build`. You may
> **not** hand-edit the version/build in either file ÔÇö regenerate
> `world-locations.json` with the extractor (which stamps the version you pass it)
> so the dataset genuinely belongs to that build. The check re-verifies the world
> dataset against fixed, known world totals, so a stale dataset wearing a new
> version number will fail.

The check also requires that `meta.channel` matches the directory, `build` is a
positive integer, and `build` is **strictly greater** than that channel's current
value (so a channel only ever moves forward). Different channels are independent.

Why a strict, isolated bundle: `en-US.json` is large and changes wholesale between
game versions, and the three files must stay in lockstep ÔÇö keeping each update to
one self-consistent channel makes data PRs trivial to review.

How to prepare one (for channel `<channel>` = `stable` or `experimental`):

1. Copy `CommunityResources/Docs/en-US.json` from a game install on the relevant
   channel to `packages/game-data-core/data/<channel>/en-US.json` (it is UTF-16 LE ÔÇö
   leave the encoding as-is; the parser handles it). Note: a patch release can ship
   a byte-identical `en-US.json` ÔÇö that is fine, the bundle still moves forward via
   `meta.json`/`world-locations.json`.
2. Write `packages/game-data-core/data/<channel>/meta.json` with the `gameVersion`,
   `build` (the in-game build number) and `channel`.
3. **Regenerate** `packages/game-data-core/data/<channel>/world-locations.json` by
   running the extractor against that build ÔÇö see
   [`tools/world-locations/README.md`](./tools/world-locations/README.md). Pass
   `GAME_VERSION`/`BUILD` so the dataset is stamped to match `meta.json`.
4. Commit **only** those three files, e.g.:

   ```bash
   git checkout -b feature/update-stable-game-data
   git add packages/game-data-core/data/stable/en-US.json \
           packages/game-data-core/data/stable/meta.json \
           packages/game-data-core/data/stable/world-locations.json
   git commit -m "chore(data): update stable game data to 1.2.3.1 (build 495413)"
   ```

5. Open a PR containing only that commit.

> **Note:** `en-US.json` is Satisfactory game data ┬® Coffee Stain Studios.
> Only the bundled channel copies under `packages/game-data-core/data/` are tracked; a local
> working copy under `game-data/` is gitignored and must not be committed.
