# FICSIT Foreman — Parser & Graph Load Technical Brief

This document describes the design of:

1. The `en-US.json` parser — **C#**, in `packages/sf-game-data/extract/sf-game-data-parse/`
2. The production-graph build — TypeScript, in `packages/sf-game-data-graph/src/`

> **The parser is C#.** It was a hand-written TypeScript parser originally; that was
> ported 1:1 to C# (validated by a golden-diff against the TS version at port time)
> and the TS parser was then retired (#159/#162). The parser now runs **offline**
> inside the `sf-game-data-extractor` tool, which writes the merged
> `sf-game-data.json` the runtime loads — it is **not** run at runtime, and the raw
> `en-US.json` is never committed. See
> [`../../docs/sf-game-data-extractor.md`](../../docs/sf-game-data-extractor.md).
> The parsing *logic* below is unchanged from the original brief; only the language,
> file paths and entry points are C#.

---

## Source File

**Path (Steam/Epic, 1.x):** `<game install>/CommunityResources/Docs/en-US.json`
**Pre-1.0:** named `Docs.json` in the same directory; handle both filenames.

Auto-generated by Coffee Stain Studios when the game is compiled; it ships with
every update and is the canonical source of truth for all game data. The extractor
reads it straight from the install at extract time; it is never committed to the
repo.

---

## Known Parsing Challenges

These are real gotchas documented by the community. Handle all of them.

### 1. UTF-16 LE with BOM
The file is **UTF-16 Little Endian** with a **byte order mark (0xFEFF)** at the
start. Decode it as UTF-16 LE (the game also occasionally ships UTF-8) and strip the
BOM before parsing. `DocsReader.DecodeBuffer` sniffs the encoding from the leading
bytes:

```csharp
var bytes = File.ReadAllBytes(filePath);
var content = DocsReader.DecodeBuffer(bytes); // UTF-16 LE / UTF-8 / BOM handling
using var raw = System.Text.Json.JsonDocument.Parse(content);
```

### 2. Top-Level Structure
The file is an array of class-group objects:

```json
[
  { "NativeClass": "Class'/Script/FactoryGame.FGItemDescriptor'", "Classes": [ ... ] },
  { "NativeClass": "Class'/Script/FactoryGame.FGRecipe'",         "Classes": [ ... ] }
]
```

Each `NativeClass` string identifies the category. Extract the short class name with
the pattern `\.([A-Za-z0-9_]+)'$` (`ClassMap.ShortNameFromNativeClass`).

### 3. No Exposed Class Hierarchy
30+ classes extend `FGItemDescriptor` (and other bases) with no hierarchy surfaced
in the file. `ClassMap.CategoryFor` maintains the `NativeClass` short name → internal
category mapping by hand:

| Short Name | Category |
|---|---|
| `FGItemDescriptor` (+ `…Biomass`, `…NuclearFuel`, `…PowerBoosterFuel`) | item |
| `FGResourceDescriptor` | resource |
| equipment / consumable / ammo / vehicle / weapon descriptors | item |
| `FGRecipe` | recipe |
| `FGSchematic` | schematic |
| anything starting `FGBuildable*` / `FGBuilding*` | building |

`FGBuildable*`/`FGBuilding*` are matched by prefix (the file has 100+ buildable
subclasses); the full item-class set is the `ItemClasses` set in `ClassMap.cs`. Any
unrecognised `NativeClass` is logged (aggregated into `parseWarnings`), never thrown.

### 4. Custom Ingredient/Product Encoding
`mIngredients`, `mProduct` and schematic `mCost` are **not** standard JSON — they use
a custom Unreal string format:

```
((ItemClass=BlueprintGeneratedClass'...Path.ClassName_C',Amount=100),(ItemClass=...,Amount=50))
```

`Normalise.ParseItemAmountList` handles it: strip the outer `(( … ))`, split on
`),(`, and regex each entry for the class name (`[^']+'[^.]+\.([A-Za-z0-9_]+)'`) and
`Amount=([0-9]+)`. Fluid scaling (÷1000) and per-minute rates are applied later, once
item forms are known.

### 5. String-Encoded Numbers
Several numeric fields are stored as strings (e.g. `"6.000000"`). `Util.GetNumber`
accepts both JSON numbers and numeric strings. Notably `mManufactoringDuration`
(seconds — match the deliberate source typo exactly) and `mCachedStackSize`.

### 6. Missing Cross-References
Build-gun recipes don't link the build descriptor to its `FGBuildable` entry. Use
the name heuristic: the product `Desc_X_C` maps to the building `Build_X_C`
(`Recipes.ExtractBuildRecipe`).

### 7. Build Costs vs Production Recipes
Distinct, kept separate:
- **Build costs** come from build-gun recipes (`mProducedIn` contains `BuildGun`) —
  what it costs to place a building; attached to the `Building`.
- **Production recipes** come from `FGRecipe` — what a building produces.

The graph models them as separate relationship types (`BUILD_COST` vs
`CONSUMES`/`PRODUCES`).

## Edge Cases — Handle, Don't Crash

- **Alternate recipes** — an item usually has several. `isAlternate` is derived from
  the class/display name. Preserve all of them.
- **Fluids and gases** — measured in m³/min, not items/min. `mForm` distinguishes
  `solid`/`liquid`/`gas`. Fluid amounts are in units of 1000 = 1 m³; divide by 1000
  for display and per-minute rates.
- **Variable-power recipes** — Particle Accelerator, Converter, Quantum Encoder vary
  power by recipe. Store the recipe-level `variablePower { min, max }`.
- **Power generators** — fuel-burn, supplemental (e.g. water) and byproduct (e.g.
  nuclear waste) per-minute rates are derived from item energy values
  (`Buildings.cs`).
- **Raw resources with no producing recipe** — iron ore, water, crude oil. Leaf
  nodes in the graph; `total_raw_inputs` terminates at them.
- **Recipes with byproducts** — multiple outputs in `mProduct`; model all as
  `PRODUCES`.
- **Items with no recipe** — valid. Surface them, don't error.

## Computed Values at Ingest Time

Computed during parsing and stored — never recalculated at query time:

```
perMinute = amount * 60 / craftTimeSeconds
```

Stored on each ingredient/product; the graph carries `perMinute` on `PRODUCES` and
`CONSUMES` so production-rate queries are trivially cheap.

## Parser Architecture (C#)

**Single responsibility:** convert the raw docs file into clean `GameData`, with no
knowledge of MCP, work orders or the foreman. All Unreal string noise is resolved
here. The project is **CUE4Parse-free** (zero dependencies), so it builds and runs
anywhere — that is what let the golden-diff gate validate it without a game install.

```
packages/sf-game-data/extract/sf-game-data-parse/
  DocsReader.cs       ← UTF-16/BOM decode + read; ParseDocsFile(path, version, build)
  GameDataParser.cs   ← orchestrator: ParseGameData(JsonElement, version, build); Warnings
  ClassMap.cs         ← NativeClass short name → category; ITEM_CLASSES set
  Util.cs             ← GetString / GetNumber over JsonElement (string-or-number tolerant)
  Normalise.cs        ← ((ItemClass=…,Amount=…)) parsing + fluid/perMinute helpers
  ClassRef.cs         ← resolves Unreal class-reference strings to short *_C names
  Items.cs            ← FGItemDescriptor subclasses + resources (form, stack, sink, energy)
  Recipes.cs          ← FGRecipe: ingredients/products/duration/machine/isAlternate; build recipes
  Buildings.cs        ← FGBuildable subclasses: power, throughput, extraction, generator fuels
  Schematics.cs       ← FGSchematic: milestones, MAM, AWESOME Shop, hard drives, unlocks
  Models.cs           ← GameData / Item / Recipe / Building / Schematic (mirror of the TS types)
  ParseJson.cs        ← serialization options (camelCase, omit-null, relaxed escaping)
```

The C# `Models.cs` mirrors the canonical **TypeScript** `GameData` types in
[`src/parser/types.ts`](./src/parser/types.ts), which the runtime consumes after
loading the merged `sf-game-data.json`. Every TS `number` is a C# `double` so JSON
number formatting matches.

## Graph Build Architecture (TypeScript)

The graph layer sits between the dataset and the MCP tools: it wraps `GameData` in
a query facade with no database — relationship queries read two precomputed
item→recipe adjacency maps, everything else reads `GameData` directly.

```
packages/sf-game-data-graph/src/
  index.ts              ← public API: initGraph(gameData: GameData): GraphDB
  graph/
    indexes.ts          ← buildGraphIndex: item→producing/consuming recipe maps
    context.ts          ← QueryContext (GameData + resolver + adjacency maps)
    resolve.ts          ← name → class-name resolver
    queries/
      items.ts          ← get_item, what_consumes
      recipes.ts        ← get_recipe, recipes_for, compare_alternates
      production.ts     ← ingredient_tree, total_raw_inputs, buildable_with
      schematics.ts     ← list_schematics, get_schematic
      buildings.ts      ← get_building, list_buildings, list_power_generators
    types.ts            ← query result types (distinct from parser types)
```

**Notes:**
- Pure in-memory, zero runtime dependencies beyond `@foreman/sf-game-data` — no
  database, no daemon, no native addon. The graph builds in sub-millisecond time.
- The facts (a recipe's ingredients/products/producedIn with `perMinute`, a
  building's build cost, a schematic's unlocks) already live fully-resolved on the
  parsed `GameData` objects; the graph just indexes item→recipe adjacency over
  them so producer/consumer lookups are O(1).
- The recursive work — ingredient-tree demand roll-up, topological ordering,
  cycle-breaking, the buildable-with fixpoint closure — runs in TypeScript.
- (Historically this layer used an embedded Kùzu graph DB; it was dropped in #243
  — the dataset is tiny (~320 recipes) and the native dependency wasn't earning
  its keep. See `docs/architecture.md`.)

## Output Types

The parser produces `GameData`. The canonical shape lives in
[`src/parser/types.ts`](./src/parser/types.ts) (TypeScript, consumed at runtime) and
is mirrored by `Models.cs`. In brief:

- **`GameData`** — `version`, `build?`, `parsedAt`, and `Record<string, …>` maps of
  `items`, `resources`, `recipes`, `buildings`, `schematics`.
- **`Item`** — `className`, `displayName`, `description`, `stackSize`, `form`,
  `sinkPoints`, `energyValue`, `isResource`.
- **`Recipe`** — `isAlternate`, `craftTime`, `ingredients`/`products` (each with
  `amount`, `perMinute`, `unit`), `producedIn`/`producedInClasses`, `inBuildGun`,
  `inWorkshop`, optional `variablePower`.
- **`Building`** — `category`, `powerConsumption`, optional
  `maxPowerConsumption`/`powerProduction`/`variablePowerProduction`/`fuels`/
  conveyor/pipe/extraction rates, and `buildCost[]`.
- **`Schematic`** — `type`, `tier`, `cost[]`, `unlocksRecipes/Buildings/Items`.

`ParseResult` wraps `{ gameData, parseWarnings }`.

---

## Version Detection

The parser does **not** sniff the version from the file or a sidecar. The
`sf-game-data-extractor` tool passes `--version`/`--build` (the in-game build) on the
command line; the parser stamps them into `GameData` and the extractor writes them at
the top of the merged `sf-game-data.json`. Absent input falls back to `'unknown'` —
never throws.

---

## Error Handling Philosophy

- **Never throw on a bad entry.** Log a warning and skip it.
- **Never silently discard data without a trace.** Skipped entries are collected into
  `ParseResult.parseWarnings`.
- **A partial parse is better than no parse.** If 95% of recipes are clean, surface
  them and warn about the rest.

## Testing & validation

- The C# parser was validated at port time by a **golden-diff**: parse the same
  `en-US.json` with both the original TS parser and the C# port and assert the
  results are structurally identical (by value). That harness was retired with the TS
  parser (#162) — its job was done.
- **Ongoing:** every bundled-data PR runs `validate-game-data`, which checks the
  dataset's `gameData` is present and non-empty and re-verifies the world totals
  against fixed known counts — so a broken parse fails CI.
- **Graph queries** (`sf-game-data-graph`) are tested against a small frozen
  `GameData` fixture (`test/fixtures/game-data.json`) for exact, controlled counts,
  and against the real bundled dataset for the power maths (known in-game values:
  Reinforced Iron Plate, Turbo Motor, Rubber + Heavy Oil Residue byproduct, Water as
  a raw leaf node).
